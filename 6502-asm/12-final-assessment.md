# Final Assessment

Across all eleven modules. Try each on paper first — no assembler, no emulator.

1. What's the one class of mistake this guide's methodology note (00-overview.md) says 6502 assembly is especially prone to, compared to Fortran, and why?
2. What does `sim65`'s `sim6502` target actually simulate — and what, specifically, does it *not* faithfully model, that this guide's two-tier verification split exists to handle?
3. Why does `ADC`/`SBC`'s carry-flag behavior make fixed-point sub-pixel movement (Module 7) "fall out for free" from ordinary 16-bit addition?
4. What's the practical difference between the C64's, NES's, and Apple II's approach to collision detection, and which two platforms share the exact same software technique?
5. Name the three interrupt vectors on a stock 6502, and which one the NES's PPU actually uses to drive per-frame game logic in a real cartridge (not this guide's bare-metal polling capstones).
6. What single routine, verified once in Module 7, was reused byte-for-byte across all three platform capstones (5, 6, 7) with zero modification?
7. Contrast the C64's 6510 I/O port, the NES's mapper concept, and the Apple II's soft switches — three completely different mechanisms solving what three different problems?
8. Why does the NES's 2A03 silently produce wrong results for BCD-based score arithmetic ported from the C64, with no assembler warning?
9. What's the shared structural trap between the C64's CIA1 (`$DC00`) and the NES's `$4016`/`$4017`, and how did this guide's capstone sidestep it on each platform?
10. Why is Apple II hi-res row 1 stored at `$2400`, nearly 1KB away from row 0's `$2000`, instead of immediately after it?
11. Why did this guide treat the VIA register table (Module 4), the interrupt vector table (Module 5), and every platform-specific hardware register (Modules 8–10) with a different confidence level than the fixed-point math and collision test (Module 7)?
12. What real, specific bug did this guide's own author encounter while building Module 7's collision test, and what fixed it?

## Answers

1. Compilers catch a large class of mistakes at compile time; a 6502 assembler mostly just checks that a valid instruction/addressing-mode combination was written, so code that assembles cleanly can still silently corrupt a register or write to the wrong address with no diagnostic at all.
2. `sim65`'s `sim6502` target is a *hosted* environment — it runs a small OS-like runtime so a C harness can call assembly and `printf` the result — not a raw simulation of a 6502 at power-on with real hardware wired to its address bus. Its own runtime occupies the interrupt vectors and top of memory, which is exactly why anything depending on real interrupt-vector mechanics or real memory-mapped hardware (a VIA, VIC-II, SID, PPU, APU) had to be documented from public sources rather than executed.
3. `ADC` naturally carries out of the low (fractional/sub-pixel) byte into the high (whole-pixel) byte whenever the sum exceeds 255 — exactly the condition "the fractional accumulator wrapped, so the on-screen position should advance by one whole pixel" — with no separate comparison or branch needed to detect it.
4. The C64's VIC-II detects sprite-sprite and sprite-background overlap in hardware, continuously, reported via `SPSPCL`/`SPBGCL` (Module 8) — no software test needed. The NES and the Apple II both have zero collision hardware and use the exact same software AABB test (Module 7's `_test_collide8`, reused unchanged in Modules 9 and 10).
5. NMI (`$FFFA`/`$FFFB`), RESET (`$FFFC`/`$FFFD`), IRQ/BRK (`$FFFE`/`$FFFF`) — standard on every 6502-family chip (Module 5). A real NES cartridge drives per-frame logic from the NMI vector specifically, since the PPU fires an NMI every vblank (Module 9's PPU session), not from IRQ/BRK.
6. `_test_collide8` — and alongside it, `_step_position` — both verified once under `sim65` in Module 7 and carried forward unmodified into Capstones 5, 6, and 7, because neither routine ever reads or writes anything but plain memory bytes; nothing about which platform's hardware surrounds them changes what they compute.
7. The 6510 I/O port (`$01`) is a bank-switcher deciding whether ROM or RAM answers at three fixed address ranges on a single, fixed 64KB map (Module 8) — it solves "how does one CPU see both ROM and RAM at the same addresses." An NES mapper is bank-switching hardware built into the *cartridge*, solving "how does a cartridge expose more ROM than the CPU's fixed address window can directly address at once" (Module 9). Apple II soft switches aren't bank-switching at all — they're addresses where the *access itself*, not a stored value, triggers a hardware action like a mode change or a speaker toggle (Module 10).
8. The 2A03 keeps the `D` (decimal) flag and lets `SED`/`CLD` set and clear it, but its BCD arithmetic circuitry was physically left out — `ADC`/`SBC` always compute in binary regardless of `D`. The assembler only validates instruction syntax, not what a specific chip's silicon does with a syntactically valid instruction, so nothing catches the mismatch.
9. Both are registers whose meaning changes depending on context or direction rather than being a single fixed input or output: CIA1 Port A (`$DC00`) doubles as the keyboard matrix's column-select output *and* joystick port 2's input, while `$4016`/`$4017` mean "strobe the controllers" on write but "read the next serial bit" on read, with `$4017` additionally meaning "APU frame-counter mode" on write specifically. Each capstone sidesteps its platform's version by using the port/direction that doesn't double up: joystick port 1 (`$DC01`) on the C64, and the documented write-then-read strobe sequence on the NES.
10. Because the hi-res page's 192 rows aren't stored linearly — they're split into three 64-row groups and interleaved within each group by a fixed hardware-defined formula (`$2000 + a*$28 + b*$80 + c*$400`), a real, documented quirk of how the Apple II's video circuitry addresses its own framebuffer, not a bug or an arbitrary design choice a programmer could route around.
11. Because they represent genuinely different kinds of claims: the fixed-point math and collision test are pure CPU logic, assembled and actually run under `sim65` with results read back and checked — verified by execution. The VIA/vector/platform-hardware tables describe real chips and real memory-mapped behavior that `sim65`'s hosted environment has no simulated hardware to exercise — cross-checked against independently-corroborated public documentation instead, and labeled as such every time, rather than presented with the same confidence as something this guide's own toolchain actually ran and confirmed.
12. An early draft of the collision test tried to pass its two operands as C function arguments to a `cl65 -t sim6502` harness, assuming cc65's calling convention would "just work" without being checked carefully — it produced garbage output. Switching to this guide's established verification pattern (named memory locations, a plain `void` routine, values set before the call, rather than trusting a register-return convention) fixed it immediately.
