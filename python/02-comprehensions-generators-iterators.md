# Module 2 — Comprehensions, Generators, and Iterators

Python's own answer to "loop and build a collection" is a language feature, not a library convention — and underneath every `for` loop this reader has written in ten prior languages, Python has one explicit, overridable protocol doing the work. Every example below is a real, verified `uv run python3` execution. Feeds Capstone 1.

## Comprehensions: building a collection in one expression

**You'll be able to:** write list, dict, and set comprehensions, including a filtering condition.

**Concept**

`[expr for item in iterable if condition]` builds a list in one expression — the direct, syntax-level equivalent of a `map`+`filter` chain from this series' JavaScript/Ruby guides, but built into Python's grammar rather than composed from separate methods. The identical shape works for dicts (`{key: value for ...}`) and sets (`{expr for ...}`).

**Example**

```python
squares = [x*x for x in range(10) if x % 2 == 0]
sq_dict = {x: x*x for x in range(5)}
sq_set = {x*x for x in [-2, -1, 0, 1, 2]}
```

Verified: `squares = [0, 4, 16, 36, 64]` (even numbers 0–8, squared), `sq_dict = {0: 0, 1: 1, 2: 4, 3: 9, 4: 16}`, `sq_set = {0, 1, 4}` — the set comprehension correctly collapses `(-2)**2` and `2**2` (both `4`) and `(-1)**2`/`1**2` (both `1`) into single set members, since a set can't hold duplicates.

## The iterator protocol: what a `for` loop actually calls

**You'll be able to:** implement `__iter__`/`__next__` on a custom class and make it work with `for`, `list()`, and every other iteration context.

**Concept**

Every `for` loop, every `list(...)` conversion, every unpacking assignment in Python runs on exactly two dunder methods: `__iter__(self)` (return an iterator — often `self`, if the object can track its own position) and `__next__(self)` (return the next value, or raise `StopIteration` when exhausted). This is the actual mechanism `range`, `list`, `dict`, and every generator (below) all implement — nothing about `for` is special-cased per type.

**Example**

```python
class Countdown:
    def __init__(self, start):
        self.current = start
    def __iter__(self):
        return self
    def __next__(self):
        if self.current <= 0:
            raise StopIteration
        val = self.current
        self.current -= 1
        return val

print(list(Countdown(5)))
```

Verified: `[5, 4, 3, 2, 1]` — `list(...)` calls `iter()` on the `Countdown` instance (which correctly returns `self`, verified separately: `iter(c) is c` reports `True`), then repeatedly calls `__next__()` until `StopIteration` is raised, exactly the loop a `for` statement runs internally.

> **Pitfall:** `__iter__` returning `self` only works correctly for an object that's inherently single-use, like this counter — a class meant to be iterated *multiple times independently* (imagine two separate `for` loops over the same `Countdown` running concurrently, each expecting its own position) needs `__iter__` to return a *fresh* iterator object each time, not `self`. Getting this wrong is invisible until something actually needs to iterate the same object twice.

## Generators: the iterator protocol, written as a function

**You'll be able to:** write a generator function with `yield`, and explain why it's lazy.

**Concept**

A function containing `yield` doesn't run its body when called — calling it returns a **generator object**, which implements `__iter__`/`__next__` automatically. Each `__next__()` call resumes the function's body from exactly where it last `yield`ed, runs until the next `yield` (or the function ends, which raises `StopIteration` automatically), and returns that value. This is the same protocol as `Countdown` above, generated by the language instead of hand-written.

**Example**

```python
def countdown_gen(start):
    while start > 0:
        yield start
        start -= 1

print(type(countdown_gen(5)))
print(list(countdown_gen(5)))
```

Verified: `<class 'generator'>`, then `[5, 4, 3, 2, 1]` — the identical output as the hand-written `Countdown` class, from roughly a third of the code, with no `__iter__`/`__next__`/`StopIteration` written explicitly at all.

**Laziness, measured directly:**

```python
gen_expr = (x*x for x in range(1000000))
print(type(gen_expr))
print(next(gen_expr))
```

Verified: creating `gen_expr` over a range of one million values is instantaneous and produces `<class 'generator'>` — nothing has been computed yet. `next(gen_expr)` pulls exactly one value (`0`), computed on demand, not looked up from a million-element structure built in advance.

```python
import sys
print(sys.getsizeof([x for x in range(100000)]))
print(sys.getsizeof((x for x in range(100000))))
```

Verified: the fully-materialized list comprehension over 100,000 integers occupies `800,984` bytes; the *identical-looking* generator expression occupies `192` bytes — a genuinely measured, roughly 4,000× difference, because the generator expression holds no elements at all, only the recipe for producing them one at a time.

> **Pitfall, verified directly and a real, common bug:** a generator is **single-use**. `list(g)` on a generator exhausts it — calling `list(g)` a second time on the *same* generator object returns `[]`, not the same values again, because there's no data sitting in memory to re-read; the generator's internal position has simply run past the end. Code that stores a generator in a variable expecting to iterate it more than once (the way a list can be iterated any number of times) silently gets nothing on the second pass, with no error to flag the mistake.

It doesn't — `Countdown` returning `self` from `__iter__` means every call to `iter()` on the same instance shares one position (`self.current`), so two separate `for` loops over the same `Countdown` object would interfere with each other's progress rather than each starting fresh from the original count.

Because a generator expression holds no materialized elements at all — only the recipe (the expression and the source iterable's current position) needed to produce the next value on demand — while a list comprehension allocates and stores every one of its 100,000 computed elements in memory at once.
