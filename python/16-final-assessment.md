## Answers

1. Python's truthiness rules treat `0` as falsy (much closer to JavaScript's rules), while Ruby's only falsy values are `nil` and `false` — `0` is truthy in Ruby. Code assuming Ruby's rules (`if count` meaning "has this been set") behaves differently the moment `count` is genuinely `0` in Python, since `if count:` there means "is this nonzero," not "is this set."
2. A function's default argument is evaluated once, at definition time, not once per call — a mutable default (like `[]`) is therefore the same shared object across every call that relies on it. `@dataclass` specifically detects a mutable default value on a field and raises `ValueError` at class-definition time, forcing `field(default_factory=...)` instead — a protection that exists only for dataclass fields, verified directly to not extend to ordinary function default arguments.
3. CPython pre-allocates and interns small integers (roughly `-5` to `256`) once, so every reference to one of those values is the identical object; integers outside that range are freshly allocated each time, even when equal. Testing with `200 + 57` is misleading because the compiler can constant-fold that literal arithmetic expression at compile time and intern the *result* within one code object — a different mechanism entirely from the runtime integer cache, which only a runtime-constructed value like `int("257")` correctly isolates.
4. That the generator expression holds no materialized elements at all, only the recipe to produce them on demand — verified directly, `800,984` bytes for a 100,000-element list comprehension versus `192` bytes for the equivalent generator expression, roughly a 4,000× difference.
5. Without it, a decorated function's `__name__` and `__doc__` (and other introspectable metadata) report the wrapper's identity instead of the original function's — verified directly (`slow_add.__name__` reporting `'wrapper'`, not `'slow_add'`, without `functools.wraps`).
6. Because an exception raised inside the `with` block is thrown directly into the generator at its `yield` point — without a `try`/`finally` wrapping that `yield`, the exception propagates straight out of the generator with nothing after `yield` ever executing, meaning any cleanup code written after a bare `yield` (no `try`/`finally`) simply never runs when an exception occurs, verified directly by removing it and observing the "releasing" message disappear.
7. `ModuleNotFoundError: No module named 'pipeline'` — the root cause was that nothing told `pytest` where the actual source file lived relative to the `tests/` directory in a `src/`-layout project; the fix required both a `pythonpath = ["src"]` config entry and importing via the full package path.
8. Summing the squares of integers up to 5,000,000 overflowed `int64`'s maximum, and `.sum()` silently wrapped around with zero warnings raised — verified directly. No individual element revealed it because every single element stayed well within `int64`'s range; only the *aggregate* (the sum across all 5 million elements) actually overflowed.
9. Because PyTorch's autograd accumulates gradients into `.grad` by default across `.backward()` calls rather than resetting them automatically — verified directly, two consecutive calls without zeroing produced `6.0` then `12.0`, not `6.0` twice. The fix is calling `.grad.zero_()` on each tracked parameter before the next `.backward()` call.
10. Because concentric circles require a circular decision boundary, and a linear model (`nn.Linear` alone, or a `Sequential` composing multiple `Linear` layers with no nonlinearity between them) can only ever express a straight-line boundary — this is a representational limitation, not a training-duration problem, verified directly by confirming a two-linear-layer model with no activation fails at the identical accuracy as a single linear layer.
11. The first measurement showed Keras roughly 221× slower than an equivalent hand-written PyTorch loop; investigating further revealed the comparison wasn't fair — Keras's default `batch_size=32` ran roughly 13× more individual training steps than PyTorch's implicit full-batch loop. Once batch size was matched, the genuine, apples-to-apples overhead was roughly 22×, not 221×.
12. GPU (MPS) acceleration helps when there's enough parallel work to amortize the overhead of dispatching to the GPU, and can be genuinely *slower* than CPU below some workload size — verified directly across three dataset sizes (CPU won at 500 samples; MPS won clearly at 20,000 and 50,000). The same shape of finding applied to Keras's overhead ratio in Capstone 5: it shrank from roughly 22× to roughly 7× as the dataset grew 125× larger, confirming that fixed per-call overhead matters proportionally less as the actual computation grows.
