# Final Assessment

Across all twelve modules. Try each on paper first — no `docker`.

1. What's the precise difference between an image and a container, demonstrated directly by writing a file inside one container and reading from a second container of the same image?
2. Why did a rebuild with no source changes take `0.199s` instead of the first build's `~10.8s`, and why did changing only `hello.txt` (in the correctly-ordered Dockerfile) still finish in a fraction of a second rather than repeating the full `~10.8s`?
3. What's the difference between `ENTRYPOINT` and `CMD`, demonstrated by Module 4's containerized `report` tool specifically?
4. What real, verified mistake did an earlier draft of this guide make while first attempting Module 6's multi-stage example, and what's the general lesson beyond "don't install PyTorch that way"?
5. What was the measured size difference between the single-stage and multi-stage versions of the identical compiled `wordcount` program, and what accounted for essentially all of it?
6. Why did a fresh container never see a file written inside a different container from the same image, and what's the one thing that changes this — verified directly with `visit-tracker`?
7. What did the Dockerfile's `VOLUME /logs` instruction guarantee, and what did it not guarantee, verified directly with two contrasting `visit-tracker` runs?
8. Why did `ping web1` fail against a container on the default `bridge` network, but succeed against an equivalently-configured container on a user-defined network?
9. What real, environment-specific mistake did this guide make while first building Module 10's web/client Compose example, and how does it connect back to an earlier module's finding?
10. What does `depends_on` in a Compose file actually guarantee, and what does it not guarantee — and what did Capstone 5's `poller` service do about the gap?
11. What survived `docker compose down`, and what additional flag would have removed it too — verified directly in Capstone 5?
12. Across this entire guide, name the one piece of computation that was verified running correctly in at least three different guises — a bare CLI tool, a containerized CLI tool, and an HTTP service — and explain why re-verifying it in each new form mattered rather than assuming it "still works."

## Answers

1. An image is a fixed, read-only template; a container is a running instance of it with its own separate, writable layer. Verified directly: a file written inside one container was completely invisible when read from a second, independent container started from the identical image — the file existed only in the first container's own writable layer, never in the shared image.
2. Docker's build cache reused every layer whose inputs were unchanged — verified directly, every instruction reported `CACHED` for the no-change rebuild. Changing `hello.txt` invalidated only the `COPY hello.txt .` layer and whatever came after it; the earlier, expensive `RUN apk add` layer's inputs hadn't changed, so it stayed cached, because the Dockerfile deliberately ordered the rarely-changing dependency step before the frequently-changing application file.
3. `ENTRYPOINT` is the command that always runs (`python3 -m code_rookie_python.cli`); `CMD` supplies default arguments to it that get replaced by anything supplied after the image name. Verified: `docker run --rm report-tool:v1` used `CMD`'s default sample file, while `docker run --rm report-tool:v1 sample_transactions.csv --min-amount 40` replaced that default while `ENTRYPOINT` still ran underneath either way.
4. Installing the shared `python/` project's full `pyproject.toml` (including PyTorch) inside a Linux container pulled in multiple gigabytes of NVIDIA CUDA libraries irrelevant to a GPU-less container, filling the host's disk. The general lesson: cross-platform dependency resolution inside a container can silently resolve to enormous, platform-specific packages that make no sense for that container's actual environment — worth checking before an unattended build runs, not just assuming a dependency list safe on one platform is safe on another.
5. `1.91GB` (single-stage, keeping the full `gcc:13` toolchain) versus `137MB` (multi-stage, only the compiled binary copied forward) — roughly a 14× difference, verified directly, accounted for almost entirely by the `gcc:13` base image and its own OS layers, none of which the already-compiled binary needs at runtime.
6. Because each container gets its own independent writable layer, entirely separate from any other container's, even one from the identical image — verified directly with `visit-tracker`, three separate `--rm`'d containers with no explicit named volume each reported `total visits: 1`, never accumulating. The one thing that changes this is explicitly mounting the same **named** volume (`-v visits_data:/logs`) across separate runs, which correctly showed the count climbing to `1`, `2`, `3`.
7. It guaranteed `/logs` was backed by a volume rather than the container's own writable layer — it did not guarantee that volume was the *same* one across separate runs. Verified directly: without an explicit `-v <name>:/logs`, each of three separate runs got its own fresh anonymous volume and each reported `total visits: 1`; with an explicit named volume, the count correctly climbed across all three runs.
8. The default `bridge` network provides IP connectivity but no automatic DNS-based name resolution between containers; a user-defined network (created explicitly, or automatically by Docker Compose) additionally provides that name resolution via Docker's embedded DNS — verified directly, the identical `ping <name>` command failed on the former and succeeded on the latter.
9. Placing the Compose project (and its bind-mounted `./webroot` directory) under `/tmp`, outside Colima's default shared-mount path — the `client` service received a directory listing instead of the real file contents, not because of any networking problem, but because Module 8 had already established that Colima only shares the home directory (`~`) by default. Moving the project under the home directory fixed it with no networking change at all.
10. `depends_on` controls start *order* only — it does not wait for the dependency to actually be ready to accept requests. Capstone 5's `poller` service included an explicit `sleep 2` before its first request specifically to cover the gap between "container started" and "HTTP server actually accepting connections," since `depends_on` alone doesn't close it.
11. The containers (`report-api`, `poller`) and the network (`report-net`) were removed by plain `docker compose down`; the named volume (`poller_logs`) survived and was verified directly by reading it back with a fresh container. `docker compose down -v` was confirmed to remove the volume too, verified directly by re-checking `docker volume ls` afterward.
12. `totals_by_category(parse_lines(...))`, computing `{"hardware": 80.0, "electronics": 45.0}` from the same four-transaction sample data — verified running correctly as a bare `uv run report` CLI invocation (`python/07-capstone-cli-tool.md`), again as a containerized CLI tool (`docker/05-capstone-containerized-cli.md`), and again as an HTTP endpoint served by `report-api` (Capstone 5). Re-verifying it each time mattered because each new form introduced its own genuinely new failure modes this guide caught directly elsewhere — a stale `ENTRYPOINT` baked into an image tag from an earlier draft, a bind-mount silently failing under Colima's home-directory restriction, a `depends_on` race condition — any of which could have silently broken the *same* underlying computation without the computation itself being at fault at all.
