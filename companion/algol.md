# Companion — ALGOL (Budokan Module 3)

**Founding paper:** Naur, P. (Ed.) (1960). "Report on the Algorithmic Language ALGOL 60." *Communications of the ACM*, 3(5), 299–314. — sourced directly from the Code Budokan Reading Workbook, Strand C.

## Historical note

ALGOL 60 is the rare language whose most successful product was never the language itself. It was never widely deployed commercially — `algol/00-overview.md` says so plainly, "nobody builds production software in it" — yet its report introduced block structure, lexical scoping, and genuine recursion into mainstream thinking about programming, and did so using a notation invented specifically to describe the language precisely: Backus-Naur Form. BNF is named for two people — John Backus (already met via FORTRAN, Module 1) and Peter Naur, who edited the ALGOL 60 report itself — and it remains the standard way a programming language's grammar gets specified, in every language reference written since, including every guide in this entire series.

The committee that produced the report also produced a real, famous, and instructive mistake: the dangling-else problem, a genuine grammatical ambiguity in how nested `if`/`then`/`else` should parse, debated for years afterward. ALGOL 68 — a later, related standard — fixed it structurally, with mandatory closing keywords (`FI`, `OD`) that make the ambiguity syntactically impossible rather than resolved by convention. `algol/04-control-dangling-else.md` verifies this contrast directly. The lesson worth carrying forward: a committee-designed specification, however rigorously written, can still ship a real ambiguity its own authors didn't fully anticipate — precisely why later languages (Pascal, C) each made their own explicit choice about how to resolve it, rather than inheriting ALGOL's own unresolved grammar.

## Reflection prompts

- BNF is the direct product of a language that itself was barely used in production. What does it mean that a specification technique outlived, and became more consequential than, the language it was invented to describe?
- The dangling-else problem is a genuine grammatical ambiguity a rigorous, formally-specified committee report still shipped. What does this suggest about the limits of formal specification generally — can a grammar be provably unambiguous and still surprise its own designers in practice?

## Short-answer questions

1. **What two things does ALGOL 60 get direct credit for in the Code Budokan history, distinct from the language's own commercial adoption?** Block structure and lexical scoping (inherited by virtually every C-family language's `{ }` scoping) and Backus-Naur Form, the grammar notation still used to specify programming languages today.
2. **What is the dangling-else problem, and how did ALGOL 68 resolve it differently from how most later languages (including C) resolved it?** A genuine parsing ambiguity in nested `if`/`then`/`else` — which `else` binds to which `if`. ALGOL 68 resolved it structurally, with mandatory closing keywords (`FI`) that make the ambiguity impossible to write; most later languages (Pascal, C) instead resolved it by convention (else binds to the nearest unmatched if), verified directly in `algol/04-control-dangling-else.md`.
3. **Why does `code-rookie`'s own ALGOL guide verify some material via `a68g` (ALGOL 68) and other material only as "documented, not executed"?** Because no maintained ALGOL 60 compiler exists to install and verify against; `a68g` is a real, actively maintained interpreter, but for a later, related, genuinely distinct 1968 standard — material shared cleanly between the two (block structure, recursion, arrays) is executed directly, while ALGOL-60-specific features ALGOL 68 doesn't have (call-by-name, Jensen's Device, the dangling-else problem itself) are sourced from the original 1960 report and well-established secondary sources instead.

## Cross-thread connection

The Budokan workbook's own master table pairs ALGOL with Turing's 1936 computability paper — "computability underpins grammar theory." The connection is precise, not decorative: BNF itself is a formal grammar system, and formal grammar theory (what can and can't be generated by a given class of grammar) sits on the same foundational logic Turing's own 1936 paper established for computation generally. A language's syntax and a language's semantics both, in the end, rest on the same 1930s theoretical ground.
