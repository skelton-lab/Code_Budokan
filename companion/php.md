# Companion — PHP (not in the original Budokan module list)

**Founding document: independently supplied, and no academic founding paper exists.** PHP began in 1994–95 as Rasmus Lerdorf's own personal set of CGI tools ("Personal Home Page Tools"), announced publicly rather than published academically — there is no Turing/Codd/Backus-style peer-reviewed paper to cite. The closest real primary-source equivalent is Lerdorf's own repeated retrospective accounts of PHP's origin across the following decades (interviews, conference talks, and the PHP manual's own "History of PHP" appendix), not a single citable document. PHP itself doesn't appear in the Budokan workbook's own module list — added to `code-rookie` after the original plan was set, the same case as `cobol.md`, `docker.md`, and `rails.md` before it.

## Historical note

PHP's own origin story is genuinely unusual among this companion's languages: it wasn't designed, in the sense Fortran or ALGOL were designed by a committee or a small team working from a stated goal — it accreted. Lerdorf built a handful of Perl scripts to track visits to his own online résumé, released them as "Personal Home Page Tools," and the tool grew feature by feature as other people started using it for their own sites, years before it had anything like a formal specification. The name itself is the clearest evidence: "PHP" originally stood for "Personal Home Page," only later becoming the recursive backronym "PHP: Hypertext Preprocessor" once the tool had outgrown its own original description.

What makes this worth taking seriously rather than treating as a punchline is that the accreted, practical-problem-first origin produced a genuinely distinctive design choice this companion's guide covers directly: code embedded inside HTML, rather than HTML generated by code. That's not a limitation PHP eventually outgrew — `php/02-embedded-templating.md` verifies it's still exactly how a `.php` file works in PHP 8.5, unchanged in its fundamental shape since the 1990s, even as the language around it added a real, modern type system (readonly properties, enums, union types — `php/06-modern-oop-types.md`) that has nothing to do with its templating origin at all. Few languages in this series show that split as cleanly: an execution model frozen since its earliest, most practical years, wrapped around a type system verified against 2026's own stable release.

## Reflection prompts

- PHP has no founding paper, no design committee, and no stated original goal beyond "track visits to my own résumé" — and yet it now runs a large share of the world's websites. What does that suggest about how much of a language's eventual reach is decided by its original design intentions, versus by what people find they can build with it once it exists?
- `php/04-request-lifecycle.md` verifies, with `getmypid()`, that PHP's own server process doesn't restart between requests — yet script-level state resets anyway, on every single one. Read that module's contrast against `javascript/07-server.md`'s own Bun-based server side by side: two real, current, actively-used execution models for "run code that responds to HTTP requests," built on genuinely opposite assumptions about what should persist. What does each model make easy, and what does each make you build by hand instead?

## Short-answer questions

1. **What does the name "PHP" originally stand for, and what does that reveal about the language's own early development?** "Personal Home Page" — before becoming the recursive backronym "PHP: Hypertext Preprocessor" later, revealing that the tool grew organically out of a specific personal use case rather than being designed from a stated general-purpose goal.
2. **What real, verified finding did `php/04-request-lifecycle.md` establish using `getmypid()`, and why does that specific tool matter to the claim?** That PHP's built-in server process does *not* restart between requests (`getmypid()` returned the identical process id across multiple requests) — yet script-level state still resets on every request regardless, proving the mechanism is PHP's own per-request execution model, not the process restarting, which a less careful investigation could easily have assumed instead.
3. **What genuinely modern PHP 8.x feature does `php/06-modern-oop-types.md` verify, that has nothing to do with PHP's own templating origin?** `readonly` properties, backed/pure `enum`s, union types, and `match` — PHP 8.1+ additions to a real, enforced type system, entirely unrelated to the HTML-embedded-code execution model that dates back to PHP's earliest years.
