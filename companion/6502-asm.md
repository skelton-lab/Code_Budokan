# Companion — 6502 Assembly (Budokan "Assembly," the substrate everything else compiles to)

**Founding papers:** von Neumann, J. (1945). "First Draft of a Report on the EDVAC." Turing, A.M. (1936). "On Computable Numbers, with an Application to the Entscheidungsproblem." — both sourced directly from the Code Budokan Reading Workbook, Strand C, where the master appendix table names them jointly for "Assembly," with the note: "All of them — assembly is what they all compile to."

## Historical note

Every other companion file in this series names a language's own founding paper — the specific intellectual leap that made that language's own distinctive idea possible. Assembly's own entry is different, and the Budokan workbook's own framing states why directly: assembly isn't one more paradigm alongside FORTRAN's imperative style or LISP's symbolic computation — it's the substrate every higher-level language in this entire companion ultimately compiles down to, and von Neumann's own 1945 stored-program architecture (instructions and data sharing the same addressable memory, executed sequentially) is the actual hardware model assembly language is a direct, human-readable notation for. Turing's own 1936 paper is paired alongside it for a reason worth stating precisely: von Neumann's architecture is one specific, physically realizable implementation of the abstract, universal model of computation Turing had already proven nine years earlier — a Turing machine's tape, head, and rule table, made concrete as memory, a processor, and a fetch-decode-execute cycle.

`6502-asm/00-overview.md`'s own promise — "build and debug real programs on bare hardware," carried through three genuinely different real historical machines (Commodore 64, NES, Apple II/IIe) — is this companion's own closest possible approach to touching von Neumann's own 1945 architecture directly, with nothing between the program and the hardware except the assembler itself.

## Short-answer questions

1. **Why does this companion pair von Neumann's 1945 EDVAC report with Turing's 1936 computability paper specifically for the Assembly entry, rather than treating assembly as one language among many?** Because assembly isn't a distinct paradigm the way FORTRAN or LISP are — it's the substrate every other language in this companion compiles down to; von Neumann's architecture is the physical hardware model, and Turing's 1936 paper is the abstract, universal computability theory that architecture is one concrete realization of.
2. **What does `6502-asm/00-overview.md` promise that gets a reader closer to von Neumann's own 1945 architecture than almost any other guide in this companion?** Building and debugging real programs on bare hardware (or a faithful simulation of it), across three genuinely distinct real historical machines, with nothing between the program and the processor except the assembler itself.
3. **What real-world constraint does `6502-asm/00-overview.md`'s own toolchain choice (the cc65 suite) solve, that a single simulator alone wouldn't?** It's the one toolchain that spans every target the guide needs — a custom bare-metal linker configuration plus official Commodore 64, NES, and Apple II/IIe targets — letting the same foundational material carry across three genuinely different real historical machines rather than one abstract simulator alone.
