# Companion — Ruby on Rails (not a language; not in the original Budokan module list)

**Founding document: independently supplied, and genuinely different in kind from this companion's other entries.** Rails has no academic founding paper — it was extracted directly from a real, working application (Basecamp) by David Heinemeier Hansson (DHH) in 2004, and first documented publicly through DHH's own blog posts and conference talks rather than a peer-reviewed paper. Thomas, D., Hansson, D.H., et al. (2005). *Agile Web Development with Rails*, Pragmatic Bookshelf — the first book-length treatment, co-written by Rails' own creator — is the closest equivalent to a founding document, and is cited here as such.

## Historical note

Rails' own origin is the sharpest possible contrast to this companion's academic founding papers: DHH didn't set out to design a web framework. He was building Basecamp, a real product, and Rails is what fell out of that work once the patterns worth reusing became visible — routing conventions, an ORM, a templating system — extracted into a separate, general-purpose framework only after they'd already proven themselves solving one company's own real problem. This is precisely why `rails/00-overview.md` frames the entire guide's own promise around *convention over configuration* as a genuinely lived design philosophy, not a slogan: `rails/01-anatomy-mvc-routing-convention.md` verifies directly that a model named `Post` maps to a `posts` table with zero explicit configuration, because Rails infers the common case from naming rather than requiring it to be stated — the same instinct that let DHH extract a general framework from one specific application's own accumulated patterns in the first place.

`rails/00-overview.md`'s own real payoff line closes a loop this companion's own `ruby.md` entry sets up directly: "the exact mechanism Rails' ActiveRecord runs on" is `ruby/06-metaprogramming.md`'s own `define_method` example, and `rails/02-scaffolding-models-migrations-dynamic-accessors.md` verifies this precisely rather than asserting it — tracing a scaffolded `Post` model's own generated `title` accessor to `ActiveModel::AttributeMethods`'s real source, confirming an evolved, batched form of the identical code-that-writes-code technique.

## Short-answer questions

1. **How did Rails originate, and what does that origin make it genuinely different from every other entry in this companion?** Extracted directly from Basecamp, a real working application DHH was building in 2004 — general-purpose patterns pulled out only after they'd already proven themselves solving one specific company's own problem, rather than designed as a general framework from the outset; unlike this companion's other entries, Rails has no academic founding paper at all.
2. **What real, verified finding does `rails/01-anatomy-mvc-routing-convention.md` establish about "convention over configuration," rather than asserting it as a slogan?** That a model named `Post` maps automatically to a `posts` database table with zero explicit configuration anywhere — verified directly, not assumed.
3. **What direct connection does `rails/02-scaffolding-models-migrations-dynamic-accessors.md` verify between Rails' own ActiveRecord and Ruby's own metaprogramming?** That a scaffolded model's generated column accessors (e.g., `title`/`title=`) are not hand-written but generated by `ActiveModel::AttributeMethods`'s real source code, using `ActiveSupport::CodeGenerator` — an evolved, batched form of the identical `define_method`-based technique `ruby/06-metaprogramming.md`'s own `make_accessor` example demonstrated by hand.
