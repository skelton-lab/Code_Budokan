# Companion — Java (not in the original Budokan module list)

**Founding document: independently supplied.** Java doesn't appear in the Budokan workbook's own module list — added to `code-rookie` after the original plan was set, the same case as `cobol.md`, `docker.md`, `rails.md`, and `php.md` before it. Unlike PHP, Java does have a real, citable primary source from its own launch: Gosling, J., & McGilton, H. (1995). *The Java Language Environment: A White Paper.* Sun Microsystems. Not a peer-reviewed academic paper in the Turing/Codd sense, but a genuine, contemporaneous technical document stating the language's own design goals in Sun's own words, at the moment of release.

## Historical note

Java began under a different name and for a different purpose than the one that made it famous. James Gosling's team at Sun Microsystems started the project in 1991 as "Oak" (renamed after a trademark conflict), targeting embedded consumer electronics — set-top boxes, interactive television — not web servers or enterprise software. That original market never materialized. What rescued the project was the World Wide Web's own sudden rise in the mid-1990s: Java's write-once-run-anywhere bytecode model, designed for a world of incompatible embedded hardware, turned out to be exactly the right shape for a world of incompatible desktop operating systems instead. "Java" itself, the shipped name, refers to the coffee, not directly to the Indonesian island it's grown on and named for — Gosling has said in interviews the name was one of several brainstormed as a replacement for "Oak," chosen because it was catchy and unclaimed, not for any deeper technical reason.

The white paper's own stated design goals are worth reading against what actually happened next: garbage collection specifically to address a named, real complaint about C/C++ (manual memory management as a source of real, common bugs), full OOP retained rather than discarded, and "architecture neutral" bytecode as the headline claim. What the white paper doesn't claim, because it couldn't yet know, is the specific role Java would go on to play in university computer-science education — succeeding Pascal and Modula-2 as the dominant introductory teaching language through the 1990s and 2000s specifically because static typing, real OOP, and garbage collection let students learn structure without also fighting manual memory management, before Python itself took over that role in the following decade. A JIT-compiled, bytecode-portable language proving fast enough in practice for that use — verified directly in this guide's own Module 9, not assumed from reputation — is the same underlying bet the white paper made about embedded hardware, just cashed in on a completely different set of machines.

## Reflection prompts

- Java was designed for a market (embedded consumer electronics) that never materialized, and became famous instead for solving a problem (cross-platform web-era software) that didn't fully exist yet when the design was made. What does that suggest about how much of a language's own eventual success is decided by its original design goals, versus by which unanticipated problem its existing design happens to fit?
- This guide's own Module 9 measured a real ~45% JIT performance gap rather than assuming "Java is fast now" from reputation. Read that finding alongside the white paper's own original claim of being "architecture neutral" — what's the actual relationship between a language being portable and a language being fast, and why doesn't one imply the other?

## Short-answer questions

1. **What was Java's original name, and what market was it originally designed for?** "Oak" — designed for embedded consumer electronics (set-top boxes, interactive television), a market that never materialized for the language.
2. **What real, verified performance finding does `java/09-jvm-bytecode-portability.md` establish about JIT compilation, rather than assuming it from Java's own reputation?** A consistent, repeatable ~45% slowdown running identical bytecode with `-Xint` (JIT disabled) versus default execution, measured across six rounds each way.
3. **What real complaint about C/C++ did the original Java white paper name directly as a design goal to solve, and what real, verified consequence of dropping the equivalent complexity does `java/03-interfaces-polymorphism.md` demonstrate?** Manual memory management as a source of common bugs, solved by garbage collection. The genuine complexity Java's own design still reintroduced in smaller form — multiple interface implementation with conflicting default methods — is demonstrated directly: `class C implements A, B` with colliding `default hello()` methods is a real compile error, forcing an explicit resolution rather than silently picking a winner.
