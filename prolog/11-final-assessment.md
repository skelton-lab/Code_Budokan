# Final Assessment

Across all ten modules. Try each on paper first — no `swipl`.

1. What's fundamentally different about querying a Prolog fact compared to calling a function in every other language in this series, and why is "no assignment statement" the single sentence that captures it?
2. What does `X = 1+2` actually bind `X` to, and what does `X is 1+2` do differently?
3. Why did `ancestor(X, jim)` return solutions in the order `pat, tom, bob` rather than some more "intuitive" order?
4. Why did `findall(L, (length(L,N), N =< 2), Ls)` exhaust the stack, and what one reordering fixed it?
5. What did switching Capstone 2's N-Queens solver from `permutation/2`+`safe/1` to `library(clpfd)` actually change, measured — give the real N=11 numbers.
6. What are the two extra arguments a DCG nonterminal compiles down to, and what earlier technique (Module 6) do they implement?
7. Why did the naive right-recursive expression grammar evaluate `10 - 2 - 3` as `11` instead of `5`, and what structural fix corrected it?
8. What's the exact, testable difference between a green cut and a red cut, using `sign_green`/`sign_red` as the reference example?
9. Why does `dislikes_wine(X)` with `X` left unbound fail outright instead of listing everyone who doesn't like wine?
10. Why does calling an undeclared, undefined predicate raise an error, while calling a `:- dynamic`-declared predicate with no current facts just fails?
11. In Capstone 4, why did the same query variable (`x`) correctly reclassify from `mammal` to `fish` mid-session with no change to `is_a/2` at all?
12. Name one genuine, unforced cross-guide callback this series' Prolog guide made to an earlier guide, and explain the actual connection (not just that both guides mention the same word).

## Answers

1. A Prolog fact isn't executed — it's stored, and a query asks whether it (or some chain of rules built from it) can be *unified* into truth, succeeding or failing with possible variable bindings rather than computing and returning a value. "No assignment statement" captures it because every other language in this series changes program state step by step; Prolog only ever asks whether a fixed database of relations can make a given statement true.
2. `X = 1+2` unifies `X` with the unevaluated compound term `+(1,2)` — no arithmetic happens. `X is 1+2` evaluates the expression on the right and binds `X` to the resulting number, `3` — verified directly (Module 1: `X == 3` fails after `X = 1+2`; Module 4: `X =:= 3` succeeds for the same `X`, since `=:=` evaluates).
3. Because the base-case clause runs before the recursive clause and immediately matched the one direct-parent fact for `jim` (`pat`), and because the recursive clause then searches `parent/2` facts in the order they were written in the source file — `tom`'s children were listed before `bob`'s, so the search reaches `tom` as an ancestor of `jim` before it reaches `bob`, purely as a consequence of file order and depth-first search, not any notion of "closeness."
4. Because `length(L, N)` with both arguments unbound backtracks through every possible length forever, and `findall` needs the whole goal to finish before it can return anything — the `N =< 2` filter, placed after the unbounded generator, never got a chance to stop the search. Reordering to bind `N` from a small range first (`between(0,2,N)`), then generating a list of that specific length, fixed it immediately.
5. Nothing about the answer changed — 2,680 solutions for N=11 either way — but the wall-clock time dropped from 108.3 seconds (permutation-and-filter) to 1.14 seconds (`clpfd`, propagating constraints as variables are declared instead of generating a complete candidate first), a measured, real, ~95x speedup on the identical problem.
6. An input list and a remainder list — the difference-list pair Module 6 built by hand first (`append_dl(A-B, B-C, A-C)`), before showing that the `-->` translation generates exactly that same pattern automatically for every grammar rule.
7. Because it recursed directly on `expr` after consuming an operator (`expr(V) --> term(T), [+], expr(V2), ...`) instead of threading an accumulator through a separate `_rest` rule — that structure parses `10 - 2 - 3` as `10 - (2 - 3)`, right-associatively. Introducing `expr_rest`/`term_rest`, threading the running total explicitly as an accumulator argument, made evaluation genuinely left-associative and produced the correct `5`.
8. A green cut only prevents wasted backtracking that would have failed anyway — every clause has its own independent guard, so removing the cut changes nothing about which answers a query produces (verified: `sign_green` gives identical results with or without its cuts). A red cut is load-bearing for correctness — a later clause (`sign_red`'s third clause) has no guard of its own and relies on the earlier cuts having already ruled out the cases it shouldn't handle; removing those cuts (verified) adds a genuinely wrong extra answer.
9. Because `\+ Goal` never binds `Goal`'s variables in the outer context — with `X` unbound, `likes(X, wine)` immediately succeeds for the first matching fact (`X = mary`), so `\+` correctly reports "this goal has a solution" and fails outright, with no binding for `X` ever produced. `\+` is only sound — only means what its name suggests — when every variable in its argument is already bound before it runs.
10. `:- dynamic` tells Prolog that a predicate having zero clauses at some point is an expected, valid state, not evidence of a missing definition — so it fails cleanly, the same way any ordinary predicate call fails when no fact matches. An undeclared, undefined predicate looks indistinguishable from a typo, which Prolog reports as an existence error rather than a silent failure.
11. Because `is_a/2`'s rules read the *current* contents of the dynamic `observed/2` database every time they run, rather than reasoning over some fixed snapshot taken once — `retract(observed(x, has_fur))` followed by `assertz(observed(x, has_gills))` genuinely changed what was true about `x` in the database between the two `diagnose/2` calls, and the same unchanged rules correctly produced a different, correct answer for the new facts.
12. Capstone 3's DCG notation and Module 6's `-->` rules are a direct callback to `algol/01-foundations-history.md`, which credits ALGOL 60's 1960 report as the origin of Backus-Naur Form and states that every language in this series describes its own grammar with a BNF descendant. Prolog is the one guide where that's literally true of the executable code, not just the documentation — a `-->` rule *is* a BNF production that also happens to run.
