**Your stated sequencing** (not file-list order): **Fortran/COBOL** → 6502 Assembly → C → **ALGOL → Pascal → Modula-2 → Simula → Smalltalk** → C++ → JavaScript → Ruby → **Prolog → SQL → Rails → PHP** → **Python/PyTorch/Keras** → **Docker**. The five historical guides sit between C and C++ deliberately, so the class/`virtual` concept lands as traceable history rather than an arbitrary C++ keyword. Pascal and Modula-2 (added after the original sequencing was set) trace ALGOL's **procedural/modular** lineage — one designer, Wirth, continuing his own line of thought — all the way through before looping back to pick up the **object-oriented** lineage from its own separate starting point (Simula, independently designed by Dahl and Nygaard in Oslo, with no connection to Wirth's work) through Smalltalk and into C++. Deliberately a lineage-first, not date-first, ordering: Simula (1967) actually predates Pascal (1970) and Modula-2 (1978) — the two threads ran concurrently, in different countries, and this sequencing finishes one before starting the other rather than interleaving two designers who were never in dialogue. COBOL (added later still) sits alongside Fortran at the very start for the same reason: Fortran (1957), COBOL (1959–60), and ALGOL (1960) are computing's three founding branches — scientific computing, business data processing, and the theoretical/structured-programming lineage this series already traces in depth — and COBOL's own guide gets the identical "verified by actually running it" treatment as everything else here, not a footnote. **Scheme** (added after COBOL) opens a fourth, entirely separate lineage this series hasn't traced yet — the Lisp family (McCarthy, 1958, older than every other language in this series) — positioned at the end of the current file list as the head of a planned **Scheme → Racket → Clojure** arc, mirroring how Pascal/Modula-2 traced Wirth's own line before Simula's fork: one lineage, followed all the way through, rather than interleaved with unrelated languages. **Racket** continues that arc directly — a deliberately fast-moving guide assuming every Scheme fundamental already in hand, covering only what Racket adds on top: contracts, `racket/class`, `syntax-parse`, and its flagship capability, building an entirely new language on Racket's own platform. **Clojure** closes the arc — persistent immutable collections, protocols and multimethods as two deliberately different polymorphism mechanisms, and the value/identity philosophy behind `atom`, all backed by real measurements rather than documentation claims. **Scheme → Racket → Clojure is now complete**, the same "one lineage, followed all the way through" discipline `pascal/`/`modula2/` established earlier in this series. **APL** (added after Clojure) joins Fortran and COBOL at the series' very start — 1962/1966, older than every other language here — the actual origin of the "whole-array operations, no explicit loop" idea `python/08-numpy-vectorization.md` covers as a modern library feature; not part of any existing lineage thread, but a direct, verified ancestor of one already in this series. **OCaml** opens a new, fifth branch — the ML lineage (ML → Standard ML/CAML → OCaml, 1996), strict and mutation-friendly. **Haskell** completes the matched pair — lazy, purity-enforced by the type system, written explicitly to be read *against* OCaml's own choices rather than in isolation: strict vs. lazy, real mutation vs. enforced purity, exceptions vs. typed failure, each guide's own capstones pointing directly at the other's. **OCaml/Haskell, the ML branch's matched pair, is now complete.** **Forth** (added after Haskell) closes out the functional-language path this series opened with `scheme/` — a sixth, genuinely distinct paradigm (stack-based/concatenative), positioned as this series' own tour of fundamentally different ways of thinking about computation: procedural, object-oriented, logic (`prolog/`), two functional lineages, and now stack-based, each verified by actually running real code, none merely described. **Julia** (added after Forth) doesn't open a new lineage — it closes a thread already running through this series: a third, genuinely distinct point on the array-oriented-programming spectrum `apl/` (the 1962-era language-level origin) and `python/`'s NumPy module (the library-level realization) already established, plus a tenth entry in the polymorphism thread via multiple dispatch as the language's own default mechanism, not an opt-in one. **Erlang** (added after Julia) closes a different thread this series left open at `prolog/`: its first implementation was literally written in Prolog in 1986, and Erlang's own pattern-matching syntax — single-assignment `=`, `[H|T]` list notation — is a direct, verified carryover, not a resemblance. Erlang's own actor-model concurrency and "let it crash" philosophy stand entirely on their own, with no prior lineage in this series. **Go** (added after Erlang) carries the C lineage forward one more generation — Ken Thompson, B's own creator and half of C's origin story (`INDEX.md`'s "ALGOL's three lineages" thread), co-designed Go decades later — and closes a second, unstated thread: `docker/`, already complete in this series, is itself written in Go. **Rust** (added after Go) closes out the compile-time-safety-foil position this series held open since `c/`/`cpp/`'s own runtime-sanitizer story: the same class of memory and concurrency bug those two guides catch with ASan/UBSan/`-race` *after* compiling and running, Rust's borrow checker and ownership rules reject *before* a binary exists at all — verified directly, not asserted, with a real data race that never compiles. It stands throughout as this series' most direct, repeatedly cross-referenced point of comparison to the guide immediately before it. **Rails** (added after Rust, chronologically the series' final guide to be written — see the note below) was repositioned in the reading pathway to sit immediately after **SQL**, not at the very end where it was originally appended. The reason is its own dependency chain, not a stylistic preference: Rails is the payoff `ruby/06-metaprogramming.md` promised directly — "the exact mechanism Rails' ActiveRecord runs on" — and simultaneously the destination `sql/`'s entire second half (joins, constraints, transactions, indexing) was quietly preparing for: ActiveRecord as a real ORM sitting directly on that relational model, not a different one. Unlike every other "added after X" guide in this list, Rails doesn't have one parent lineage — it has two, Ruby *and* SQL — so it doesn't belong appended to either individually (the way the 6502 hardware ports extend `6502-asm/`, or PyTorch/Keras extend `python/`); it belongs read as its own guide, immediately after both prerequisites are fresh, cashing in two threads several earlier guides deliberately left open rather than letting them go cold across another dozen unrelated languages.

**PHP** (added after Rails, the series' twenty-ninth guide) closes this cluster the same way Rails did — its own case for inclusion was never a new paradigm (another dynamically-typed, C-family-influenced, multi-paradigm scripting language, territory `python/`/`ruby/`/`javascript/` already cover three times over) but a specific, real historical footprint: the only major web-native, template-embedded scripting language in this series, still literally how a `.php` file works today. It reads right after `rails/`, not merely near it, for the same dependency reason Rails itself was repositioned — Capstone 2 directly rebuilds `javascript/07-server.md`'s own `/api/users` API and finds the opposite result (state resets on *every* request, not just a restart, verified with `getmypid()` proving the server process itself never restarted at all), and Capstone 4's PDO work is a direct, named callback to what Rails' own ActiveRecord automates. Both prerequisites — JavaScript's server chapter and SQL's relational material — are exactly as fresh here as they were for Rails, and PHP's own overview names the connection to Rails specifically, not just generically to "the database guides."

## The cross-guide threads

### Polymorphism (the long one — twelve guides)

The single thread every guide's "Beyond This Guide" or capstone section keeps returning to:

1. **C** — a struct with a hand-assigned function pointer (`Shape` capstone), built by hand, no language support at all.
2. **Simula** (1967) — `Virtual` procedures. The actual historical origin, documented in that guide's Module 4. Bjarne Stroustrup has stated directly (his own book, his HOPL-II paper) that he carried this exact term into "C with Classes."
3. **C++** — `virtual`, a compiler-generated version of what C built by hand. Module 3.
4. **JavaScript** — the prototype chain. Traced (Module 3) through **Self** (1987) back to the Smalltalk lineage — the *other* branch of the tree, not Simula's.
5. **Smalltalk** — message sends resolved entirely at runtime, no shared type ever required. Module 7 works through, mechanically, *why* Stroustrup's performance requirements made him choose Simula's static discipline over this model.
6. **Ruby** — duck typing. Module 4 states plainly (backed by the Smalltalk guide's own Module 5) that this is a direct descendant of Smalltalk's object model, not an independent rediscovery.
7. **Racket** — `racket/class`, `define/override` plus explicit `super` calls (Module 4/Capstone 2) — a Lisp-family entry in the thread, verified with a mixed list of `employee%` subclasses dispatching correctly with no `cond`/`case` anywhere in the calling code, the same "one message, many behaviors" shape as every entry above it, in a language none of the first six share any lineage with.
8. **Clojure** — two deliberately separate mechanisms, not one: `defprotocol`/`extend-type` (Module 3), single-dispatch like every prior entry but retroactively extensible onto types you don't own, verified directly by extending a built-in Java type; and `defmulti`/`defmethod` (Module 4), dispatch on an arbitrary computed value or a *combination* of multiple arguments' types together — something no other language in this thread, including Racket's `class*`, can express at all.

9. **Haskell** — type classes (`class`/`instance`, `haskell/04-type-classes.md`), resolved by the type checker at compile time (like OCaml's functors) but with implicit, inferred call-site selection (like Clojure's runtime dispatch feels to a caller) — a genuinely third position in the design space, precisely distinguished from both. Capstone 2 also surfaced the real cost static typing exacts here: a heterogeneous list needed an explicit existential wrapper (`AnyShape`) and a language extension, something every dynamically-typed language in this thread got for free.
10. **Julia** — multiple dispatch (`julia/01-foundations-types-dispatch.md`), the only entry in this entire thread where dispatch on *every* argument's type is the language's own default mechanism for *every* function, including its own built-in operators — not an opt-in alternative to ordinary function calls the way Clojure's multimethods are (`julia/02-comparing-dispatch-mechanisms.md`), and not limited to a single receiver the way every OOP-style entry above it is. Capstone 1's nine-method collision system is this thread's clearest demonstration yet of a problem single dispatch genuinely cannot express without an artificial workaround.
11. **Go** — interfaces (`go/03-interfaces-implicit-satisfaction.md`), satisfied implicitly like Ruby's duck typing (no declaration connecting a type to an interface) but checked at compile time like OCaml's functors or Haskell's type classes — verified directly with a real, precise compiler error naming the exact missing method. Capstone 2 showed this working across a boundary neither side controls: the standard library's own `fmt.Stringer` interface, detected and used automatically on types written years after `fmt` itself.
12. **Rust** — traits (`rust/06-traits-dynamic-dispatch.md`), a genuine third position next to Go's own entry directly above it: nominal like Ruby/Smalltalk's explicit method definitions rather than structural like Go's interfaces (a type needs a real `impl Trait for Type` block, verified directly that a matching method alone isn't enough), but still dynamically dispatched through `dyn Trait` at runtime, the same underlying mechanism as a Go interface value. Capstone 2 verified the honest limit of the analogy: unlike Go's `fmt.Stringer`, detected automatically on any matching type, Rust's own `std::fmt::Display` requires the identical explicit `impl` as any other trait — confirmed directly with a real `E0277` when a type without one was printed with `{}`.

Also: Ruby's `nil`/`true` being real singleton instances of `NilClass`/`TrueClass` is the identical design as Smalltalk's `UndefinedObject`/`True` — Matz has stated Smalltalk directly shaped Ruby's object model (Ruby Module 1).

### Operator overloading (three guides, three syntaxes)

Fortran's `interface operator(+)` → C's explicit "we don't have this, here's why" (Module 5) → C++'s `operator+` (Module 5) → Ruby's `def +(other)` — same underlying idea (register a function as an operator's implementation), compared directly in each guide.

### C as the universal FFI target

Fortran's `iso_c_binding` signpost and C's "C as FFI target" signpost (Module 8) point at each other directly — the same boundary from both sides.

### Verification-discipline as a running bit

Every guide's testing session ties back to the same idea in a different syntax: Fortran's `check()`/`error stop`, the 6502 guide's memory-plus-C-harness technique, C/C++'s sanitizer-verified examples, JavaScript's `bun test`, Ruby's `minitest`, Prolog's `plunit` (folded into its Capstone 1 module), Python's `pytest` (Module 6, including a real `ModuleNotFoundError` the guide hit and fixed live), Rust's `cargo test` (`rust/13-testing-cargo-test.md`, built directly into the toolchain, its own capstone extension closing out Capstone 4), Rails' own `minitest` (`rails/10-testing-fixtures-model-controller-integration.md`) — the identical framework `ruby/07` already used, not a new one, with `ActionDispatch::IntegrationTest` closing this series' verification-discipline thread on its own final guide. Referenced explicitly in most of the "Files/testing" modules.

### Wirth's own lineage: ALGOL → Pascal → Modula-2

Three guides, one designer's continuous line of thought. `pascal/01-foundations.md` opens with Wirth sitting on the ALGOL 68 committee, disagreeing with its complexity, and designing Pascal independently instead. `pascal/02-control-procedures-parameters.md` traces the dangling-else problem through a third guise — ALGOL 60 has it undocumented, ALGOL 68 fixes it structurally, Pascal keeps it and resolves it by convention (the same convention C later adopted). `modula2/00-overview.md` and its Module 2 pick up directly where `pascal/10-beyond-this-guide.md` left off: Pascal's missing module system, named there as the specific reason Wirth built Modula-2 next. `modula2/05-coroutines.md` closes the loop forward into this series' *next* guide, comparing Modula-2's `TRANSFER` directly against Simula's independently-designed `detach`/`resume` — real convergence, not influence, stated precisely as such.

### ALGOL's three lineages

Not two branches from ALGOL 60 — three, running through this series independently and never crossing. **Wirth's branch**: ALGOL 60 → (ALGOL 68 committee, rejected) → Pascal → Modula-2, one designer's continuous line of thought, above. **Dahl & Nygaard's branch**: ALGOL 60 → Simula (1967), built directly as an ALGOL 60 extension for simulation work, its `class` construct traceable to an ALGOL block with one change (`simula/02-classes-objects.md`) — then on to Smalltalk and, via Stroustrup, C++. **The Cambridge/Bell Labs branch**: ALGOL 60 → CPL (Cambridge/London, ~1963, explicitly an ALGOL-family language) → BCPL (Martin Richards, ~1966) → B (Ken Thompson, ~1969, built for early Unix on the PDP-7) → C (Dennis Ritchie, ~1972, adding a real type system to B). C's `{ }` block structure is this branch's inherited ALGOL shape. Three teams, three countries, three unrelated motivations (a workstation OS, physical simulation, fitting a compiler on a PDP-7) pulling the same 1960 root in different directions — including the quiet irony that C and Simula are siblings by way of ALGOL, but neither language's design ever looked sideways at the other.

### BNF made executable

The ALGOL guide (`algol/01-foundations-history.md`) credits ALGOL 60's 1960 report as the origin of Backus-Naur Form and notes every language in this series describes its own grammar with a BNF descendant. Prolog's Capstone 3 (DCGs, `prolog/06-dcg-grammars.md` and `07-capstone-dcg-parser.md`) is the one guide where that's true of the running code, not just the documentation — a `-->` rule is a BNF production that also executes.

### COBOL's `RECORD KEY` and SQL's `PRIMARY KEY`: the same problem, decades apart

`cobol/08-indexed-files-search.md` names this directly: an indexed COBOL file's `RECORD KEY` (fetch, update, or delete one specific record fast, by a unique identifying value, guarded by `INVALID KEY`) and SQL's `PRIMARY KEY` solve the identical underlying problem — decades apart, one via direct file-system-level indexing, the other via a full relational query engine built on top of the same basic need. `cobol/10-copybooks-subprograms.md`'s C-interop capstone extension is also a direct, real instance of this series' C-as-universal-FFI-target thread, made unusually easy by GnuCOBOL's own C-based compilation pipeline — and its fixed-decimal `PIC` arithmetic (`cobol/02-arithmetic-picture-clauses.md`), verified side by side against Python's `0.1 + 0.2` float-approximation result, is the same "the numeric type quietly did something the value on paper didn't expect" family as `python/08-numpy-vectorization.md`'s `int64` overflow, from the opposite direction: COBOL's decimal fields were designed from 1959 specifically to avoid that exact class of error for money.

### Predicate logic, twice: Prolog and SQL

`sql/00-overview.md` opens by naming this directly rather than treating it as coincidence: E.F. Codd's 1970 relational model and Robert Kowalski's logic programming both trace to predicate logic. Datalog — named as a signpost in both `prolog/10-beyond-this-guide.md` and `sql/11-beyond-this-guide.md` — is the language actually sitting at the intersection: relational-algebra-complete and a restricted Prolog simultaneously. SQL's recursive CTEs (`sql/07-window-functions-cte.md`) are also called out as the direct structural echo of Prolog's `ancestor/2` (`prolog/02-recursion-backtracking-lists.md`) — anchor/`UNION ALL`/recursive-member is base-clause/recursive-clause in different syntax.

### Erlang, Prolog's own child: a lineage, not a resemblance

`prolog/` was never just a thematic ancestor for `erlang/` — it's the literal implementation language of Erlang's first prototype (Ericsson, 1986), and `erlang/01-foundations-pattern-matching.md` verifies the connection directly rather than asserting it: Erlang's `=` is Prolog's own unification, not assignment — binding an already-bound variable to a different value fails with a real error, exactly like a Prolog variable refusing to unify with a conflicting value. `erlang/02-multi-clause-functions-guards.md` carries this further — `[H|T]` is Prolog's own list notation, borrowed directly, and multi-clause functions are the same base-case/recursive-case shape as a multi-clause Prolog predicate. `erlang/03-capstone-pattern-matching.md` reimplements `prolog/03-capstone-kinship.md`'s exact family-tree problem, verified to produce identical answers — with the honest limit stated directly: Prolog's automatic backtracking search has no equivalent Erlang inherited, only its pattern-matching syntax and semantics did.

### Backtracking, built-in vs. built-by-hand: Prolog and Scheme

`prolog/02-recursion-backtracking-lists.md` gets backtracking as a language-level feature, free, with no code written for it. `scheme/08-capstone-continuations-control-flow.md` builds the identical capability from scratch — McCarthy's `amb` operator, implemented entirely from `call/cc` and mutable state — and solves N-Queens with it, the same generate-and-test structure `prolog/05-capstone-search.md` used. The honest comparison is the point: Scheme's version is materially more code and materially easier to get subtly wrong (a real `let`-vs-`let*` bug surfaced during its own verification), not because Scheme is worse at search, but because Prolog's execution model *is* backtracking search by default, where Scheme instead offers `call/cc` — general enough to build backtracking (or exception handling, or generators) yourself, at the cost of building it yourself.

### Array-oriented programming, three points on one spectrum: APL → NumPy → Julia

`apl/01-foundations-arrays-shape.md` is the actual 1962-era origin: whole-array operations with no explicit loop, built directly into the language's own core design, no marker needed. `python/08-numpy-vectorization.md` is the same instinct as a *library*, layered onto an otherwise ordinary, loop-capable language decades later. `julia/06-broadcasting-dot-syntax.md` is a third, genuinely distinct point: a real, dedicated language feature (`.` broadcasting) that's deliberately opt-in and visible at every call site — neither APL's "everything is elementwise, always" nor NumPy's "vectorized under the hood." `julia/07-capstone-array-computation.md` closes the loop concretely: the exact same ten-value dataset and statistics computation as `apl/03-capstone-statistics-engine.md`, run independently in a language built fifty years later, producing the identical mean, variance, and standard deviation.

### One computation, three guises: Python → Docker

`docker/`'s entire running capstone thread reuses `python/`'s own `report` tool and `pipeline.py` logic directly, rather than inventing a disconnected example — the identical `totals_by_category(parse_lines(...))` computation is verified independently as a bare `uv run report` CLI invocation (`python/07-capstone-cli-tool.md`), again as a containerized image (`docker/05-capstone-containerized-cli.md`), and a third time as an HTTP endpoint served from inside a two-container Compose project (`docker/11-capstone-compose.md`) — the same `{"hardware": 80.0, "electronics": 45.0}` result, checked fresh each time rather than assumed to "still work" once wrapped in a new layer.

### Two guides setting up a third: `ruby/` and `sql/` → `rails/`

`ruby/06-metaprogramming.md` names this destination directly, mid-guide, before `rails/` existed: "the exact mechanism Rails' ActiveRecord runs on." `rails/02-scaffolding-models-migrations-dynamic-accessors.md` cashes that promise in, verified rather than asserted — tracing `Post`'s own generated `title`/`title=` accessor pair to `ActiveModel::AttributeMethods#define_attribute_methods`'s real source, confirming it's an evolved, batched form of the identical technique `ruby/`'s own `make_accessor` example built by hand with plain `define_method`. `sql/`'s entire second half — joins, constraints, transactions, indexing — is the relational layer `rails/04-associations-has-many-belongs-to.md` and `rails/05-validations-presence-uniqueness-custom.md` sit directly on top of, confirmed at the database level, not just the Ruby level: a raw SQL insert bypassing ActiveRecord's own validations entirely is still rejected by a real `FOREIGN KEY`/`UNIQUE` constraint, `sql/09-transactions-constraints.md`'s own material, verified once more from the ORM side rather than assumed to still hold once a framework sits on top of it.

### Real bugs found *while building these guides*, kept in as teaching material

Not hypothetical gotchas — things that actually broke during verification, left in deliberately:

- Fortran: `v(::-1)` does not reverse an array (empty section instead) — the original error this whole project corrected — and, separately, a `pure` abstract-interface pattern that compiles standalone but fails the moment anything `use`s it (a real gfortran bug, Capstone 2).
- C: `return &local` compiles with only a warning, then "works" in a test run until it silently doesn't (Module 2).
- C++: a hand-written move constructor missing the null-out step reproduces a real double-free under ASan (Module 6).
- ALGOL: `a68g` requires uppercase `BEGIN`/`END` by default (stropping), not a bug but a real, undocumented-until-you-hit-it convention.
- Pascal: a subrange type (`WorkDay = Mon..Fri`) looks like a compile-time-enforced guarantee — verified directly, it's only enforced for literal constants by default; the identical violation delivered through a variable compiles and runs with no error at all unless `-Cr` (runtime range checking) is passed explicitly (Module 4). Separately, Free Pascal's default mode was found to actually *allow* pointer arithmetic on typed pointers (`p := p + 1` compiles cleanly) — contradicting the reasonable assumption "Pascal doesn't have pointer arithmetic," which turned out to be true only of the ISO standard (verified via `-Miso`), not of this guide's own anchored toolchain's default behavior (Module 8).
- Smalltalk: `aFloat printString` raises `ZeroDivide` in this exact `gst` build — worked around with `rounded`/`truncated` throughout.
- Prolog: `findall(L, (length(L,N), N =< 2), Ls)` doesn't return three short lists — it exhausts SWI-Prolog's 1GB stack outright, because `length/2` with both arguments unbound generates lists of every length forever and the `N =< 2` filter never gets a turn (Module 4). Fixed by binding `N` from a bounded range first.
- SQL: the guide's own first-draft claim that a deleted `INTEGER PRIMARY KEY` value is never reused turned out to be only half true — re-verifying directly showed SQLite reuses a freed rowid *only* when it was the table's current maximum, not for a mid-table gap. Caught before shipping (Module 1). Separately, Capstone 2's indexing extension found `EXPLAIN QUERY PLAN` reporting *no* change at all after adding an index to a four-row join — the honest, correct result at that scale, contrasted directly with Module 5's real 130× speedup on 1,000,000 rows.
- Python: summing the squares of 5,000,000 integers via NumPy's default `int64` silently overflowed and produced a wrong total, with zero warnings raised — every individual array element was correct; only the aggregate was wrong (Module 8). Separately, Capstone 5's first Keras-vs-PyTorch timing comparison showed a dramatic, real 221× gap — investigating rather than reporting it revealed an unmatched `batch_size` was responsible for most of it; the corrected, fair comparison was a still-real but far smaller ~22×.
- Docker: an early Module 6 draft tried installing `python/`'s full shared `pyproject.toml` (PyTorch included) inside a Linux container, which pulled in multiple gigabytes of NVIDIA CUDA libraries irrelevant to a GPU-less container and filled the host machine's actual disk mid-build — fixed by switching the multi-stage example to a compiled-C-program comparison instead (`1.91GB` → `137MB`, ~14×, measured cleanly). Separately, Capstone 1's first verification pass was run against a `greeter:v1` image tag that still held a stale `ENTRYPOINT` baked in from a superseded draft of the Dockerfile — caught by inspecting the image's actual `Config` directly rather than trusting that the shown Dockerfile and the tagged image still matched, and the whole capstone was rebuilt and re-verified cleanly before shipping.
- COBOL: the guide's own first attempt to verify fixed-form column-72 truncation behavior got the wrong answer — a test line that looked like it disproved the "columns past 72 are ignored" folklore, only because the "past 72" content in that test was still, by miscount, sitting at column ≤72. Caught by redoing the test with exact per-column character inspection rather than trusting a visual line-length count; the folklore turned out to be true all along (Module 1). Separately, verified directly that an unguarded `COMPUTE`/`ADD` overflowing a `PIC 9(3)` field doesn't crash or leave the value unchanged — it silently keeps only the low-order digits that fit (`999 + 500` → `499`), a real, dangerous finding for money fields (Module 2).
- Scheme: Capstone 2's `amb` backtracking search, first written with simultaneous `let` bindings, produced a technically-correct but unexpected result (`(5 5)` instead of the leftmost `(1 9)` for a two-sum search) — traced to `let`'s unspecified binding-evaluation order conflicting with `amb`'s sequential trap-door mechanism; switching to `let*` fixed it (Capstone 2). Separately, a direct, measured contrast: a tail-recursive sum to 10,000,000 completed correctly with no stack growth, while an otherwise-identical naive-recursive version aborted with "maximum recursion depth exceeded" at the exact same depth — proper tail calls verified as a real, load-bearing guarantee, not an assumed one (Module 5).
- Racket: the flagship custom-`#lang` capstone genuinely failed on its first attempt — a language providing its own functions but never re-exporting `#%app`/`#%datum` produced `no #%app syntax transformer is bound` the moment a program tried to call one; fixed by explicitly re-providing the core forms the language actually needed (Module 8/Capstone 4). Separately, verified directly rather than assumed: Typed Racket does **not** work out of the box against this guide's own anchored `minimal-racket` toolchain — `#lang typed/racket/base` fails outright, requiring a separate `raco pkg install` (Module 10).
- Clojure: not a caught mistake but a deliberately measured pair of real numbers backing claims most guides would only assert — 1000 `assoc` operations on a 1,000,000-element persistent vector took 0.72ms, versus ~4953ms for 1000 genuine full copies of the same vector (Module 1); and 10 real JVM threads each incrementing a shared counter 10,000 times lost 79,181 of the expected 100,000 updates when run against unprotected mutable state, versus an exact match through `atom`/`swap!` (Capstone 3). Separately, `0.10` combined via ordinary arithmetic produced `0.18999999999999995`, not a clean `0.19` — the same IEEE 754 float-precision finding this series already verified independently in COBOL and Scheme, now confirmed a fourth time (Capstone 2).
- APL: GNU APL hangs indefinitely with zero output under the default `C` locale — every example in this guide required `LC_ALL=en_US.UTF-8` set explicitly, discovered by direct testing, not documentation (Module 0). Separately, dfn guard syntax (`condition:result`, the standard way most APL dialects express conditionals inside a function) failed outright — `Illegal : in immediate execution` — tested directly against this guide's own invocation mode, even for the simplest possible case; reported as a limitation of that specific invocation mode, not generalized beyond what was actually tested (Module 4). A third, smaller catch: a first-draft variable name, `nearest-per-store`, parsed as an attempted subtraction between three undefined names, since APL identifiers can't contain hyphens (Capstone 2).
- OCaml: `side_effect "a" 1 + side_effect "b" 2` printed `evaluating b` before `evaluating a` — verified directly that OCaml's strict evaluation guarantees every subexpression runs, but not in what order, contradicting the left-to-right assumption a reader coming from most of this series' other languages would likely make (Module 1). Separately, this guide reported one topic — OCaml 5's effect handlers — as explicitly **unverified**, not claimed and not silently omitted, when a direct attempt at a minimal working example produced a syntax error (Module 10), the same honesty standard normally applied only to confirmed bugs, extended here to an unconfirmed claim instead.
- Haskell: a function typed `Int -> Int` (claiming purity) embedded `putStrLn "sneaky!"` via `seq` — it compiled, but running it never printed anything at all, proving a sharper truth than "pure functions can't do IO": an `IO` action is inert data, permitted to exist inside pure-typed code, that only actually executes if connected to `main`'s own action chain (Module 8). Separately, Capstone 1's traced Fibonacci-primes composition reproduced Clojure's own identical "asking for zero elements computes zero elements" result (`clojure/09-capstone-infinite-sequences.md`), reached through opposite defaults — Clojure's explicit opt-in laziness versus Haskell's laziness-by-default for every expression — verified as the same observable outcome by two genuinely different mechanisms.
- Forth: two genuine, live-caught bugs during capstone construction, both kept in deliberately. Capstone 1's `HYPOT-SQ`, defined as `SQUARE SQUARE +`, produced `259` instead of `25` for a 3-4-5 triangle — the second `SQUARE` squared the *first* `SQUARE`'s own result, not the original second operand, since an operator only ever touches whatever currently sits on top; fixed with explicit `SWAP`s. Capstone 3's `WEIGHTED-SUM` failed outright with a real `Stack underflow` — its first term carried a trailing `+` left over from mechanically copying the same pattern onto every line, with nothing yet computed to add it to. Separately, verified directly: calling an under-supplied word produces a raw runtime error with zero compile-time arity checking anywhere in the language (Module 1).
- Julia: `Base.return_types` proved, not merely suggested, that `balance / 1` — mathematically a no-op — genuinely destabilized an entire function's type inference (`Union{Float64, Int64}` instead of a single concrete type), measured at an ~11× real performance cost (Capstone 2); a related but structurally different instability trigger measured separately at ~6700× (Module 4) — both real, both kept in, with the differing magnitude itself stated as a finding rather than smoothed into one round number. Separately, Capstone 3's statistics computation, run against the identical ten-value dataset `apl/03-capstone-statistics-engine.md` used, reproduced the exact same mean, variance, and standard deviation independently.
- Erlang: a first-draft claim in Module 2 — that reordering `classify`'s three list-shape patterns (`[]`, `[_]`, `[_, _ | _]`) would break `classify([1])` — turned out to be wrong on direct testing, since those three patterns are mutually exclusive by construction; caught before shipping and replaced with a genuinely overlapping pair (`describe_num(N)` before `describe_num(0)`) that the compiler itself flagged with a real "cannot match" warning. Separately, Capstone 3's own construction hit a genuine race condition in an early design (a globally registered process name re-registered too soon after a crash) — resolved by redesigning the capstone to avoid the race entirely rather than patching it with longer sleeps, with the reasoning kept in the guide's own overview.
- Go: not a caught mistake but a deliberately two-sided measurement, run specifically to avoid a one-sided concurrency claim — the exact same worker-pool program measured ~5× *slower* concurrently on cheap per-item work (checking primality below 200,000) and ~3.3× *faster* concurrently on expensive per-item work (checking primality of numbers near 100 billion), both real, both kept in (Capstone 1). Separately, `result, _ := safeDivide(10, 0)` was verified to compile cleanly and run with zero warning of any kind, silently producing `0` as if it were a legitimate result (Module 5) — the concrete version of Go's own most common real criticism, checked directly rather than repeated from reputation.
- Rust: this guide's own Capstone 4 first draft, `fn parse_record(line: &str) -> Result<ParsedRecord, String>`, compiled and ran correctly but produced a real compiler warning — `hiding a lifetime that's elided elsewhere is confusing` — caught during verification and fixed directly from the compiler's own suggestion (`Result<ParsedRecord<'_>, String>`), a working program that still had something real worth fixing. Separately, three real transcription risks were caught and corrected during this guide's own verification pass, not left to chance: a panic message quoting a stale filename/line from an earlier scratch file (Module 4), a trait-not-satisfied error whose exact text depends on which other types are in scope (Module 6), and a data-race example that, in an earlier draft capturing a loop variable, produced four compile errors instead of the three actually claimed (Capstone 3) — each re-verified against the *exact* code shown before shipping. Separately again, not a mistake but a genuinely two-sided finding of its own: Capstone 3's concurrent prime-counting workload measured *faster* with threads in both a cheap and an expensive regime (~2.4× and ~2.9×), the opposite direction from `go/`'s own measured result for the analogous workload — both real, neither smoothed to agree with the other.
- Rails: a bare `rails new` genuinely fails its own automatic `bundle install` step — `Errno::EACCES` writing a plugin shim into Homebrew's read-only system gem directory — traced via a full backtrace to `Gem::InstallerUninstallerUtils#regenerate_plugins_for`, fixed permanently with a one-time global `bundle config set --global path 'vendor/bundle'` (Module 0). Separately, `bin/rails generate authentication` run against this guide's own `--minimal`-anchored toolchain produces a genuinely broken `PasswordsController` — no `create` action, and its own auto-generated test references a `PasswordsMailer` class that was never created — because `--minimal` comments out `require "action_mailer/railtie"` directly in `config/application.rb`, and the password-reset flow assumes it exists (Module 7). Separately again, this guide's own Capstone 4 construction caught two genuine mistakes in its own test-writing, live: a cascading-delete test that expected `Comment.count` to change by `-1` but got `-2`, because it reused a shared fixture that already had a comment attached (not accounted for); and a Capstone 2 `curl` verification that grabbed the wrong CSRF token from a page with two forms, genuinely rejected by Rails 8's `per_form_csrf_tokens` scoping, resolved by using the specific form's own token rather than any valid token on the page.
