# Module 2 — Embedded Templating

PHP's actual origin, still exactly how a `.php` file works today: code embedded directly in HTML, not HTML generated by code. Feeds Capstone 1.

## `<?php ?>` and `<?=` inside real HTML

**You'll be able to:** write a file that's mostly HTML with PHP embedded at exactly the points that need it, and explain why this is PHP's own founding idea, not a framework convention.

**Concept**

A `.php` file is HTML by default; `<?php ... ?>` opens a region of PHP code, and everything outside those tags is sent to the browser verbatim, unmodified. `<?= $expr ?>` is shorthand for `<?php echo $expr; ?>` — the single most common thing you'll write in a template. This is genuinely different from `javascript/06-dom-events.md`'s own model, where HTML lives in `.html` files and JavaScript manipulates the DOM afterward, as a separate, later step — PHP's template *is* the file; there's no separate manipulation phase.

**Example**

```php
<?php
$name = "Ada";
$items = ["apples", "bread", "milk"];
?>
<!DOCTYPE html>
<html>
<body>
  <h1>Hello, <?= htmlspecialchars($name) ?>!</h1>
  <p>You have <?= count($items) ?> items.</p>
</body>
</html>
```

> **Pitfall:** `<?= $userInput ?>` without `htmlspecialchars()` is a real, live cross-site-scripting hole — anything a user can control that reaches an unescaped `<?= ?>` gets executed as HTML/JavaScript in every other visitor's browser. Verified directly: `?name=<script>alert(1)</script>` through a bare `echo $name` prints the literal, executable `<script>` tag; through `htmlspecialchars($name)` it prints the harmless text `&lt;script&gt;alert(1)&lt;/script&gt;`. Escape output by default, not as an afterthought.

**Practice**

- Build a small page that loops over a list of three products and prints each one's name and price, using `<?= ?>` for each value.
- Deliberately leave one interpolation unescaped, view the page source, and confirm you can see why that would be dangerous with real user input.

## The alternate control-structure syntax

**You'll be able to:** write `if`/`foreach` blocks using PHP's `:`/`endif;`/`endforeach;` syntax, and explain why it exists specifically for templates.

**Concept**

Alongside the normal `{ }`-brace control structures, PHP has an alternate syntax — `if (...): ... endif;`, `foreach (...): ... endforeach;` — built specifically so control flow reads cleanly when it's interleaved with HTML rather than wrapping it in braces that visually clash with markup. It's not a different mechanism, just different punctuation for the same `if`/`foreach`; real, current, and genuinely more common in templates than the brace form.

**Example**

```php
<?php
$items = ['apple', 'banana', 'cherry'];
$loggedIn = true;
?>
<ul>
<?php if ($loggedIn): ?>
  <li>Welcome back</li>
<?php endif; ?>
<?php foreach ($items as $i => $item): ?>
  <li><?= $i ?>: <?= htmlspecialchars($item) ?></li>
<?php endforeach; ?>
</ul>
```
```html
<ul>
  <li>Welcome back</li>
  <li>0: apple</li>
  <li>1: banana</li>
  <li>2: cherry</li>
</ul>
```

> **Pitfall:** mixing the two syntaxes in the same block (`if (...): ... }`) is a real syntax error, not a style warning. Pick one per block — this guide uses alternate syntax specifically inside templates, brace syntax everywhere else (a convention, not a rule PHP itself enforces).

**Practice**

- Add a conditional that shows "No items" using `else:` when the list is empty, and verify it renders correctly for both an empty and non-empty array.

## `$_GET` and `$_POST`: form data as superglobals

**You'll be able to:** read query-string and form-submitted data through PHP's superglobal arrays, with the null-coalescing operator guarding against a missing key.

**Concept**

`$_GET` holds query-string parameters (`?name=Ada` becomes `$_GET['name']`); `$_POST` holds form-submitted body data for a `POST` request with `Content-Type: application/x-www-form-urlencoded` or `multipart/form-data`. Both are ordinary arrays, available in every request without any setup — "superglobal" means accessible from any scope, including inside a function, without an explicit `global` declaration. `$_GET['name'] ?? 'default'` — the null-coalescing operator — is the standard way to read a key that might not be present, without a separate `isset()` check first.

**Example**

```php
<?php
$name = $_GET['name'] ?? 'stranger';
echo "Hello, " . htmlspecialchars($name) . "!\n";
```

Verified against a running server with real requests:
```
$ curl "localhost:8936/get-test.php"
Hello, stranger!

$ curl "localhost:8936/get-test.php?name=Ada"
Hello, Ada!
```

> **Pitfall:** `$_GET['name']` (no `??`) throws an `Undefined array key` warning — not a fatal error, but real, visible noise — the moment a request arrives without that parameter. `??` isn't optional politeness; it's the difference between a page that degrades gracefully and one that logs a warning on every visit missing an optional parameter.

**Practice**

- Build a small form (a plain HTML `<form method="post">`) that submits a name and a favorite color, and a PHP script that reads both from `$_POST` and displays them back, escaped.
- Confirm with `curl -X POST -d "name=Ada&color=blue" ...` that the same script works without a browser at all.

## Progress check

1. What does `<?= $expr ?>` expand to, and why is it worth knowing the long form?
2. What's the real, live consequence of printing `$_GET['name']` with a bare `echo` instead of through `htmlspecialchars()`?
3. Why does PHP have an alternate `:`/`endif;` control-structure syntax at all, given brace syntax already exists?
4. What does `??` actually check, and what does it return when the left side doesn't exist?

### Answers

1. `<?php echo $expr; ?>` — the short form is just syntax sugar. Knowing the long form matters because `echo` itself can take multiple comma-separated arguments and isn't limited to a single expression the way `<?= ?>` reads at a glance.
2. Any HTML/JavaScript in the unescaped value executes in the browser exactly as if the page's own author had written it — a live, verified cross-site-scripting vulnerability, not a theoretical one; confirmed directly with `?name=<script>alert(1)</script>` producing a literal, executable `<script>` tag.
3. It exists specifically so control flow reads cleanly when interleaved with HTML markup — brace syntax visually clashes with surrounding tags in a way `:`/`endif;` doesn't, which is why alternate syntax is the more common real-world choice inside templates specifically.
4. It checks whether the left-hand expression is `null` or doesn't exist (an unset array key, for instance) — `$_GET['name'] ?? 'stranger'` returns `'stranger'` exactly when `$_GET['name']` is missing or `null`, without throwing the `Undefined array key` warning a bare access would.
