# Final Assessment

Across all eleven modules and four capstones. Try each on paper first — no `php`.

1. What's the actual difference in output between `'$name'` and `"$name"`, and what does `declare(strict_types=1)` change that has nothing to do with that?
2. Why is `["a", "b", "c"]` and `["x" => 1, "y" => 2]` the same type in PHP?
3. What real, live consequence was verified from printing `$_GET['name']` with a bare `echo` versus through `htmlspecialchars()`?
4. What did `getmypid()` prove across two requests to the same `php -S` server, and what did the counter example prove *despite* that?
5. In Capstone 2, what specifically happened to a user added via `POST /api/users` by the time the very next `GET /api/users` ran — and how does that differ from `javascript/07-server.md`'s own version of the identical API?
6. What's the precise boundary of what `readonly` actually protects, verified directly against a property holding an array?
7. Name the two concrete ways `match` differs from `switch`, both verified directly.
8. Character for character, what does `WHERE name = '$name'` become when `$name` is `' OR '1'='1`, and what did that produce against the real two-row `users` table?
9. What's the actual mechanism that let `$_SESSION` survive across requests when Module 4 proved plain script variables can't?
10. Why is comparing two `password_hash()` outputs with `===` always wrong, verified directly?
11. In Capstone 4, what two specific things were confirmed to survive a full server restart, and why did they survive when Capstone 2's array didn't?
12. What does PSR-4 autoloading eliminate the need for, and what real, verified failure mode does a mismatched namespace/directory produce?
13. Name the one genuine, unforced cross-guide connection this guide's overview made, and state precisely what it claims — not just that two guides mention the same word.

## Answers

1. Single-quoted strings never interpolate — `'$name'` prints the four literal characters `$name`; double-quoted strings interpolate variables directly. `declare(strict_types=1)` is unrelated to interpolation entirely — it makes *that file's* own typed function calls throw a real `TypeError` on a type mismatch instead of silently coercing the argument, and only for that file.
2. PHP has exactly one compound type — the array — serving as both an ordered list and a string-keyed map, distinguished only by which keys happen to be present, not by separate type declarations the way most other languages in this series (Python's `list`/`dict`, Ruby's `Array`/`Hash`) draw that line.
3. Any HTML/JavaScript in the unescaped value executes in the browser exactly as if the page's own author wrote it — verified directly with `?name=<script>alert(1)</script>` producing a literal, executable `<script>` tag through bare `echo`, versus the harmless escaped text `&lt;script&gt;alert(1)&lt;/script&gt;` through `htmlspecialchars()`.
4. `getmypid()` returned the identical process id across both requests, proving the underlying server process never restarted. The counter example still reset to `1` on every single request regardless — proving PHP discards all script-level state at the end of every request independent of the process's own lifetime, the real mechanism, not "the server restarted."
5. The added user was gone — the very next `GET` returned only the original two seeded users, not three. `javascript/07-server.md`'s Bun-based version keeps a `POST`-added user in every subsequent `GET` until the process itself restarts, because its array lives in one long-running process's memory; PHP's version loses it on the next request regardless of restart, because PHP tears down script-level state after every request.
6. `readonly` blocks reassigning the property binding itself from outside the declaring scope after construction — verified directly, reassigning `$p->x` threw `Error: Cannot modify readonly property`. It does not make anything reachable *through* that property deeply immutable; a `readonly array` property's own contents remain mutable through code with access to that array.
7. `match` compares with strict `===` equality, never loose `==`; and it's an expression that evaluates directly to a value rather than a statement requiring `break` per branch. Also verified: an unmatched value with no `default` arm throws `UnhandledMatchError` rather than silently doing nothing.
8. `WHERE name = '' OR '1'='1'` — the attacker's `'` closes the string literal early, and the remainder becomes real SQL syntax: an always-true `OR` condition. Against the real table, `vulnerableLookup()` returned *both* rows, including `bob_admin`'s `is_admin` flag — the entire table, not just the intended lookup's match.
9. Session data is written to disk (or another configured store) at the end of each request and read back at the start of the next, keyed by a cookie the browser resends — it never depended on in-process memory surviving between requests, so Module 4's per-request teardown doesn't touch it at all.
10. `password_hash()` randomizes the salt on every call, so the same password produces a different hash string each time — verified directly, hashing the identical string twice produced two different outputs, both of which `password_verify()` correctly matched against the original password. `===` would incorrectly report even the correct password as a mismatch whenever the hash was generated at a different time than the one being compared against.
11. Both a task row (in the file-backed SQLite database) and the logged-in session itself survived a full `php -S` restart, verified directly by killing and restarting the server process between requests and confirming the same cookie still resolved to the same logged-in user with the same task list intact. They survived because both live in real, external, file-backed storage — a database file and PHP's own file-based session store — neither of which depends on the server process's own memory the way Capstone 2's plain array did.
12. Manually `require`-ing every individual file a script depends on. A mismatched namespace-to-directory mapping doesn't error at `composer install` time — it fails later, the first time something actually tries to instantiate the class, with a "Class not found" error that doesn't obviously point back to the mapping as the cause.
13. PDO's prepared-statement/parameter-binding work (Module 8) is literally what `rails/`'s own ActiveRecord automates — the same real relationship `rails/`'s own overview named for `sql/`'s material, now traced one layer down: Rails' Capstone 4 tested that a raw SQL insert bypassing ActiveRecord's validations was still blocked by a real database-level constraint; this guide built, by hand, the exact prepared-statement layer that made that safety possible in the first place.
