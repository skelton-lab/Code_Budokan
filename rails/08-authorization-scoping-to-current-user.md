# Module 8 ŌĆö Authorization: Scoping Records to `current_user`

Authentication answers "who is this?" Authorization answers a genuinely separate question: "is this specific user allowed to do this specific thing?" Rails 8's native generator gives a real, verified answer to the first question (Module 7); this module builds and verifies the second from scratch, since no generator provides it automatically. Feeds Capstone 3.

## `Current`: a real per-request singleton

**You'll be able to:** explain what `ActiveSupport::CurrentAttributes` provides, and use `Current.user` inside a controller.

**Concept**

`Current` (generated by the auth generator, `app/models/current.rb`) is an `ActiveSupport::CurrentAttributes` subclass ŌĆö a thread-safe, per-request singleton, automatically reset between requests. `Current.session` is set once, when a request is authenticated; `Current.user` is delegated from it.

**Example**

```ruby
class Current < ActiveSupport::CurrentAttributes
  attribute :session
  delegate :user, to: :session, allow_nil: true
end
```

`delegate :user, to: :session, allow_nil: true` means `Current.user` calls `Current.session&.user` ŌĆö a real, working `Current.user` in every controller and view, for the entire duration of one authenticated request, with no need to pass a user object through every method call manually.

> **Pitfall / gotcha:** `Current` is explicitly reset between requests by Rails itself ŌĆö it is *not* a global, application-wide singleton the way a plain Ruby class-level `@@variable` would be. Treating `Current` as safe for anything longer-lived than a single request (background jobs, for instance) is a real, common mistake; those contexts need their own way of establishing "who is this for."

**Practice**

- Add a `helper_method :current_user` (aliasing `Current.user`) to `ApplicationController`, and use `current_user.email_address` directly in a view.
- Verified directly: is `Current.user` available inside a model (e.g., `Post`), or only inside controllers and views? Try referencing it from a model method and see what happens.

## Locking down and opening up: `allow_unauthenticated_access`

**You'll be able to:** state what `ApplicationController`'s default authentication requirement is, and use `allow_unauthenticated_access` to opt specific actions out.

**Concept**

`ApplicationController`'s `include Authentication` adds `before_action :require_authentication` ŌĆö running on **every action, in every controller that inherits from it**, by default. This is a real, significant consequence worth being deliberate about: generating authentication doesn't just add a login page, it locks the entire application behind one, immediately, unless something explicitly opts out.

**Example**

```ruby
class PostsController < ApplicationController
  allow_unauthenticated_access only: %i[ index show ]
  # ...
end
```

Verified directly, both directions:

```bash
curl -o /dev/null -w "%{http_code}\n" http://localhost:3000/posts       # 200
curl -o /dev/null -w "%{http_code}\n" http://localhost:3000/posts/new   # 302, redirected to login
```

Guests can read (`index`, `show`, explicitly opted out); creating, editing, and destroying still require a real session, by the controller's own unmodified default.

> **Pitfall / gotcha:** verified directly, and a genuinely easy thing to forget ŌĆö a *new* controller, added later, requiring **zero** extra configuration to require login, is exactly right for most actions but silently wrong for any action meant to be public. This project's own `CommentsController`, added in Module 4/Capstone 2 before authentication existed at all, was never updated with `allow_unauthenticated_access` ŌĆö the moment Module 7's generator ran, commenting silently started requiring login too, without a single line changed in `CommentsController` itself. Verified directly: `Filter chain halted as :require_authentication rendered or redirected` in the server log, the first time an unauthenticated comment attempt was tested. Not a bug ŌĆö a real, worth-understanding consequence of "secure by default."

**Practice**

- Confirm directly, via `curl` or a browser, that `POST /posts/1/comments` genuinely redirects to the login page for a guest, and explain, from `ApplicationController`'s own code, precisely why ŌĆö without `CommentsController` itself containing any authentication-related code at all.
- Decide, and justify in one sentence: should commenting require login in this app? Implement `allow_unauthenticated_access only: [:index]` on `CommentsController` if you decide comments should be readable but not postable by guests (there's no `CommentsController#index` in this guide's own scope ŌĆö this is a genuine design decision left to you).

## Ownership: a `before_action` that actually checks

**You'll be able to:** write a `before_action` that verifies a record belongs to the current user, and redirects if not.

**Concept**

Authentication alone (Module 7) proves *someone* is logged in ŌĆö it says nothing about whether *this specific* logged-in user owns *this specific* record. A dedicated `before_action`, run only on the mutating actions, closes that gap explicitly.

**Example**

```ruby
class PostsController < ApplicationController
  before_action :set_post, only: %i[ show edit update destroy ]
  before_action :require_ownership, only: %i[ edit update destroy ]

  def create
    @post = Current.user.posts.build(post_params)
    # ...
  end

  private
    def require_ownership
      unless @post.user == Current.user
        redirect_to posts_path, alert: "You can only edit your own posts."
      end
    end
end
```

Two details worth being precise about: `create` builds the new post through `Current.user.posts.build(...)` ŌĆö the association itself, not a bare `Post.new` with a manually-assigned `user_id` ŌĆö making it structurally impossible to accidentally create a post owned by someone else. And `require_ownership` runs *only* on `edit`/`update`/`destroy`, deliberately not on `show` ŌĆö reading someone else's post is fine; changing it isn't.

> **Pitfall / gotcha:** `require_ownership` redirects rather than raising ŌĆö a real, deliberate choice, verified directly to still block the attack (Capstone 3 confirms this with two genuinely separate users), but worth distinguishing from an approach that raises `ActiveRecord::RecordNotFound` or a `403 Forbidden` instead. A redirect-with-alert is friendlier for a real user who followed a stale link to a post they used to own; a hard 403 is more honest about a genuine access-control violation. Either is defensible ŌĆö the redirect approach is what this capstone verifies, not the only correct answer.

**Practice**

- Change `require_ownership` to `raise ActiveRecord::RecordNotFound` instead of redirecting, and verify directly what HTTP status a browser or `curl` request now receives.
- Verify directly, with two real registered users: does `require_ownership` alone (with no CSRF token issue at all) block an edit attempt, or does the earlier-verified CSRF per-form-token scoping (Capstone 1) end up doing part of the blocking? Isolate the two by using a token you know is genuinely valid for the attacking user's own session.

4. What's the structural difference between `Post.new(user_id: some_id)` and `Current.user.posts.build(...)`, regarding the possibility of accidentally creating a record owned by the wrong user?

**Answers**

1. `Current.session`, via `delegate :user, to: :session, allow_nil: true`. `Current` (an `ActiveSupport::CurrentAttributes` subclass) is automatically reset by Rails between requests ŌĆö it's a per-request singleton, not an application-wide one.
2. `before_action :require_authentication`, added via `include Authentication` in `ApplicationController` ŌĆö applying to every action, in every controller that inherits from `ApplicationController` (which is all of them, by Rails' own convention), unless a specific controller or action opts out with `allow_unauthenticated_access`.
3. It silently started requiring login for every action, including `create` and `destroy` on comments ŌĆö verified directly via the server log (`Filter chain halted as :require_authentication rendered or redirected`) the first time an unauthenticated comment attempt was tested, with zero lines of `CommentsController` itself ever touched.
4. `Post.new(user_id: some_id)` accepts whatever `user_id` is passed ŌĆö a real risk if that value ever comes from unchecked input. `Current.user.posts.build(...)` builds through the association itself, making the owning user structurally fixed to `Current.user`, with no `user_id` parameter to accidentally trust from the request at all.
5. No ŌĆö `require_ownership` is scoped with `only: %i[ edit update destroy ]`, deliberately excluding `show`; reading another user's post is allowed, only changing it is blocked.
